# Python Code Style

按下面规则生成 Python 代码，并优先兼容用户当前项目已有风格；如果项目已有写法与本 skill 不冲突，保持一致。

## 核心规范

- 优先做最小必要改动，不顺手重构整条链路，不主动调整项目结构。
- 尽量避免重复代码，但不要为了消除少量重复强行拆成很多小函数或抽象层。
- 没必要单独提炼的方法，直接保留在一个完整的大方法中，优先保证调用链清晰。
- 变量名、函数名、类名要直接表达业务含义，不使用过度缩写。
- 关键分支、边界处理、数据转换要补充简洁注释，不写逐行翻译式注释。

## Python 代码约定

- 遵循 Python 3 常见风格和 PEP 8，保持代码整洁、缩进统一、导入明确。
- 公共函数、核心业务函数优先补充类型标注；如果项目本身几乎不用类型标注，则保持现有风格。
- 异常处理只保留必要分支，不为了“看起来安全”滥用 `try/except`。
- 判断逻辑优先直写，避免为了简单流程引入额外设计模式。
- 能直接返回就直接返回，避免无意义的中间变量和多层嵌套。

## FastAPI 与接口代码

- 路由层只负责接收参数、调用 service 或核心逻辑、返回结果。
- 具体业务判断、数据拼装、状态处理放在 service 或业务方法中。
- 请求体、响应体优先使用 Pydantic 模型描述。
- 新增接口时保持函数命名清晰，参数和返回结构稳定。

## 测试代码约定

- 如果用户要求生成测试方法，实现逻辑必须非常简单，方便直接运行和理解。
- 不要使用 `add_argument` 这类命令行参数解析方式包装测试入参，直接使用普通变量。
- 测试优先覆盖主流程、关键分支和明显边界，不写过度复杂的构造逻辑。
- 除非用户明确要求，禁止你自动测试

## 生成代码时的执行方式

- 先理解现有文件风格，再补全或修改代码。
- 如需新增代码，优先沿用原有命名、入参组织和返回风格。
- 如需补注释，优先解释“为什么这样做”以及边界含义。
- 如需在“减少重复”与“保持直观”之间取舍，优先选择更直观、改动更小的实现。
- 生成的代码应可直接落地，避免只写空壳、TODO 或伪代码。
- 使用框架内已有的日志工具
- 缩进使用 tab
