# Daily Report

## 适用场景

- 用户说“日报”“今日日报”“根据今天的提交写日报”。
- 用户要求按今天或指定日期的 Git commit 记录整理工作内容。
- 用户要中文工作日报，而不是提交代码或生成 commit message。

## 核心原则

- **日报是给老板看的**，老板关心：今天解决了什么问题 / 完成了什么需求 / 修复了什么缺陷，然后才是大致怎么解决的。
- **技术实现细节（如类名、方法名、配置拆分）不是重点**，除非是重大技术攻克。

## 处理流程

1. 先确认当前目录是否是 Git 仓库；如果不是，说明无法从提交记录生成日报。
2. 默认使用本机当前日期作为“今天”，按本地时区统计当天 00:00 到现在的提交。
3. 用 `git log --since="00:00" --name-status --pretty=format:"%h%x09%an%x09%ad%x09%s" --date=iso-local` 获取当天提交。
4. 如果用户指定日期，按该日期的 00:00 到次日 00:00 查询。
5. 如果没有提交记录，直接说明今天没有查到提交，并给出空日报或简短说明。
6. 生成日报时合并相近提交，按业务/模块归类，避免逐条机械复述 commit message。
7. **从对话历史中自动提取**：在今日对话历史中(`用 ~/.codex/sessions/$(date +%Y/%m/%d)/*.jsonl`)查找与当前git提交的文件相关的对话。
8. **如果对话历史中找不到**：根据 git 改动，自行分析总结出了什么问题, 并标注 "归纳: xxxxx"


## 写作规则（强约束）

- 日报不是 commit 摘抄，也不是技术架构说明。
- 每一条必须回答两个问题：
  1. 今天解决了什么问题？
  2. 为了解决这个问题，完成了什么改动？
- 优先写“已完成的事情”，不要写“看起来很高级的技术名词堆砌”。
- 禁止大量使用以下空话，除非后面有明确对象或结果：
  - 完善
  - 优化
  - 规范
  - 升级
  - 重构
  - 增强
  - 全面接入
- 如果只是代码层面的整理，没有直接业务结果，要明确降级表述为：
  - “代码重构：修改xxx功能时便于添加新的功能，不影响现有功能...”
  - “代码重构：提高可读性...”
- 不要把“DTO、配置拆分、拦截器、解码器、常量”单独写成一条日报，除非它直接支撑了某个明确问题的解决。
- 同类提交必须合并，最多输出 3~5 条。
- 如果无法从 commit message 和相关对话中判断真实业务背景，允许写“归纳”，但“归纳”条目最多 2 条。
- 最终结果必须像人写的工作日报，不能像代码目录说明。

## 输出格式

- 为了解决问题`xxx`，改动内容 `xxx`。
- 为了解决问题`xxx`，改动内容 `xxx`。
- 其他：`xxx`
