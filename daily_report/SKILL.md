# Daily Report

## 适用场景

- 用户说“日报”“今日日报”“根据今天的问题和改动写日报”。
- 用户要求按今天或指定日期的 Codex 对话历史和 Git 改动整理工作内容。
- 用户要中文工作日报，而不是提交代码或生成 commit message。

## 核心原则

- **日报是给老板看的**，老板关心：今天解决了什么问题 / 完成了什么需求 / 修复了什么缺陷，然后才是大致怎么解决的。
- **技术实现细节（如类名、方法名、配置拆分）不是重点**，除非是重大技术攻克。

## 处理流程

1. 先确定日报日期：默认使用本机当前日期；如果用户指定日期，使用用户指定日期。
2. **优先读取当天对话历史**：默认查找 `~/.codex/sessions/$(date +%Y/%m/%d)/*.jsonl`；指定日期时按对应 `YYYY/MM/DD` 目录查找。
3. 从对话历史中提取用户当天输入的问题、需求、报错、改动意图，以及 assistant 最终完成的任务摘要；不要把完整对话逐条复述。
4. 从每个 session 的 `session_meta.payload.cwd` 提取会话启动目录，并对每个 cwd 执行 `git -C "$cwd" rev-parse --show-toplevel` 反推 Git 仓库根目录。
5. 如果 `session_meta.payload.cwd` 不存在或不能反推仓库，可辅助使用 `turn_context.payload.cwd`、`event_msg.payload.cwd`；仍无法确认仓库时，该 session 只作为问题/过程来源。
6. 对反推出的 Git 仓库去重后，逐个读取 Git 改动确认项目真实变化：
   - 当天提交：`git log --since="00:00" --name-status --pretty=format:"%h%x09%an%x09%ad%x09%s" --date=iso-local`
   - 指定日期：按该日期 00:00 到次日 00:00 查询。
   - 工作区改动：必要时查看 `git status --short`、`git diff --name-status`、`git diff --cached --name-status`。
7. Git 改动可能包含用户手动修改和 Codex 修改。
8. 用“对话里遇到的问题和意图”作为日报问题来源，用“Git 改动”作为完成成果来源，交叉印证后合并相近事项。
9. 如果对话历史不存在或没有相关内容，再根据 Git 改动自行分析总结出了什么问题，并标注 `归纳: xxxxx`。
10. 如果对话历史和 Git 改动都没有有效信息，直接说明今天没有查到可生成日报的内容，并给出空日报或简短说明。


## 写作规则（强约束）

- 日报不是 commit 摘抄，也不是技术架构说明。
- 每一条必须回答两个问题：
  1. 今天解决了什么问题？
  2. 为了解决这个问题，完成了什么改动？
- 优先写“已完成的事情”，不要写“看起来很高级的技术名词堆砌”。
- 禁止大量使用以下空话，除非后面有明确对象或结果：
  - 完善
  - 优化
  - 规范
  - 升级
  - 重构
  - 增强
  - 全面接入
- 如果只是代码层面的整理，没有直接业务结果，要明确降级表述为：
  - “代码重构：修改xxx功能时便于添加新的功能，不影响现有功能...”
  - “代码重构：提高可读性...”
- 不要把“DTO、配置拆分、拦截器、解码器、常量”单独写成一条日报，除非它直接支撑了某个明确问题的解决。
- 同类提交必须合并，最多输出 3~5 条。
- 如果无法从对话历史和 Git 改动中判断真实业务背景，允许写“归纳”，但“归纳”条目最多 2 条。
- 最终结果必须像人写的工作日报，不能像代码目录说明。

## 输出格式

- 为了解决问题`xxx`，改动内容 `xxx`。
- 为了解决问题`xxx`，改动内容 `xxx`。
- 其他：`xxx`
- 说人话,一定要输出非技术人员看得懂又能体现工作成果和难度的日报
